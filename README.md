## 它是什么

一个 AI 技能（Skill）。装在你的 AI 助手里，给它加一个能力：**提示词优化**。

你随口说一句需求，它帮你诊断缺了什么，然后输出三个版本的优化提示词——从能用、到好用、到专业级，你挑一个。

不需要学 prompt engineering，不需要背模板，甚至不需要知道什么是"角色设定"或"约束条件"。你只管说人话，它负责把人话翻译成 AI 能听懂的人话。

## 怎么用

装好之后，直接跟你的 AI 助手说：

```
优化这个提示词：你是个设计师，帮我画个 logo
```

它会先告诉你缺了什么（诊断），然后给出三个版本（极简 / 规范 / 发布级），最后说明改了什么、为什么改。

除了"优化这个提示词"，以下说法也能触发：

- `帮我优化提示词`
- `改写提示词`
- `improve my prompt`
- `make this prompt better`

也可以直接把一段原始需求丢给它，不加任何前缀——如果它判断你在做提示词相关的事，会自动进入优化模式。

## 效果

### 示例 1：从一句话到三个版本

**输入：**

```
你是个设计师，帮我画个 logo
```

**极简版** — 一句话，粘贴即用

> 你是品牌视觉设计师，专精科技类初创公司的 logo 设计。请为一家 AI 代码助手工具设计 logo，风格简洁现代，适合在小尺寸下使用。

**规范版** — 日常工作够用

> 你是品牌视觉设计师，拥有 8 年以上科技类初创公司 logo 设计经验。请为一家 AI 代码助手工具设计 logo。
>
> 要求：风格简洁现代，在 16px favicon 和 200px 展示图下都清晰可辨。避免通用元素（齿轮、灯泡、代码括号）。提供 2-3 个方向的概念草图，每个方向附带设计理念说明。

**发布级版** — 高价值任务 / 写进文档里给他人用

> 你是品牌视觉设计师，拥有 8 年以上科技类初创公司 logo 设计经验，熟悉从概念到落地的完整品牌视觉流程。请为一家 AI 代码助手工具设计 logo。
>
> 工具的核心用户是中小型软件公司的开发者，定位是"开发者的搭档而非替代者"。要求风格简洁现代、有辨识度但不过度设计，在 16px favicon 到印刷品尺寸下都保持清晰。避免科技行业过度使用的元素（齿轮、灯泡、代码括号、大脑轮廓）。
>
> 提供 2-3 个方向的概念草图，每个方向说明设计理念和目标情绪。如果某个方向需要放弃，也说明原因——我会从对比中做判断。

### 示例 2：模糊需求也能处理

**输入：**

```
帮我写个产品介绍
```

**极简版：**

> 你是资深科技产品文案专家。请为 AI 代码助手工具撰写产品介绍，目标受众是中小型软件公司技术负责人。突出核心功能和 ROI。

**规范版：**

> 你是一名资深科技产品文案专家，拥有 10 年 SaaS 产品营销经验。请为新上线的 AI 代码助手工具撰写产品介绍。
>
> 目标受众：中小型软件公司的技术负责人。要求：突出核心功能，强调投入产出比，使用专业但易懂的语言。输出结构由你根据内容自行组织。

**发布级版：**

> 你是一名资深科技产品文案专家，拥有 10 年 SaaS 产品营销经验，熟悉开发者工具市场的用户痛点和购买决策链。请为新上线的 AI 代码助手工具撰写产品介绍。
>
> 目标受众：中小型软件公司的技术负责人，他们关心效率提升、团队 onboarding 速度和实际代码质量改善。要求：突出核心差异化功能，用真实场景说明价值，强调可量化的投入产出比。使用专业但易懂的技术语言，避免过度营销腔。输出结构由你根据内容自行组织——怎么讲清楚就怎么讲。

### 示例 3：专业场景

**输入：**

```
你作为玻璃面板高级工程师，这是一份报告，请你深度分析里面存在的问题，技术性问题
```

**极简版：**

> 你是玻璃面板高级研发工程师，专精质量分析与工艺缺陷诊断。请审查以下技术报告，从数据可靠性、工艺合规性、技术逻辑严密性三个角度分析问题并标注严重程度（高/中/低）。

**规范版：**

> 你是玻璃面板行业的高级研发工程师，拥有 10 年以上产品质量分析与工艺缺陷诊断经验。以下是一份技术报告，请以资深评审者视角进行深度技术审查。
>
> 审查重点：数据与结论是否有充分对应关系；工艺参数和判定结果是否与相关标准一致；技术逻辑是否严密——因果推断是否排除了合理替代解释。对每个问题说明问题所在、为什么是问题、严重程度（高/中/低），按严重程度降序排列。只列有实质技术依据的问题，不凑数。

**发布级版：**

> 你是玻璃面板行业的高级研发工程师，拥有 10 年以上产品质量分析与工艺缺陷诊断经验，熟悉强制性标准测试流程和常见失效模式。以下是一份技术报告，请以资深评审者视角进行深度技术审查。
>
> 审查重点：数据与结论是否有充分对应关系；工艺参数和判定结果是否与相关标准一致；技术逻辑是否严密——因果推断是否排除了合理替代解释，是否存在循环论证或跳跃性结论。
>
> 对每个问题说明问题所在、为什么是问题、严重程度（高/中/低），按严重程度降序排列。不凑数——没有实质技术依据的问题不列。报告整体质量高时，先肯定做得好的部分，再指出可改进之处。

## 三个版本有什么区别

| | 长度 | 场景 | 特点 |
|--|------|------|------|
| **极简版** | 1-3 句 | 自己用、快速搞定 | 角色到位、任务清晰，复制粘贴就能跑 |
| **规范版** | ≤10 句 | 日常工作、发给同事 | 加了上下文和约束，完整但不冗余 |
| **发布级版** | ≤20 句 | 高价值任务、写进文档 | 上下文丰富、质量标准明确、有负向约束 |

每个版本都是独立可用的——不会出现"在上面版本基础上添加以下内容"这种事。

## 原理

### 7 维诊断

拿到你的原始输入后，它会逐项检查 7 个维度：

| 维度 | 检查什么 | 常见问题 |
|------|---------|---------|
| **清晰度** | 目标明确吗？ | "帮我写个东西"——什么东西？ |
| **上下文** | 有背景信息吗？ | 没说受众是谁、什么场景 |
| **约束** | 有边界吗？ | 没说格式、长度、语调、范围 |
| **角色** | 指定了专家身份吗？ | "你是个设计师"——什么领域？什么级别？ |
| **结构** | 输出格式明确吗？ | AI 不知道你想要表格还是文章 |
| **示例** | 有参考样本吗？ | 没有 Few-shot 示例引导风格 |
| **推理引导** | 有思考方向吗？ | 复杂任务缺少分析框架 |

只列出真正有缺陷的维度——没问题的不会硬凑。

### 三层结构

根据任务复杂度选择不同的骨架：

- **Tier 1**（骨架）：角色 + 任务 + 格式 → 简单问答、一次性任务
- **Tier 2**（标准）：+ 上下文 + 约束 → 分析、创作、多步骤任务
- **Tier 3**（高级）：+ 示例 + 推理引导 → 复杂工作流、高价值输出

### 克制优先

这是这个技能和其他提示词工具最大的区别。

**大多数工具的做法：**
- 给你塞模板、填表格
- 加一堆"请按以下格式输出：## 概述 / ## 分析 / ## 建议"
- 规定字数、章节数、列名

**这个技能的做法：**

| 做 | 不做 |
|----|------|
| 给 3-5 条核心原则 | 塞 15 条 checklist |
| "根据内容自行组织" | "按以下格式输出" |
| 定义 WHAT（审什么） | 规定 HOW（怎么想） |
| 信任模型判断力 | 规定死板的章节顺序和字数 |
| 10 行说清楚 | 80 行事无巨细 |

原因很简单：**大模型不是表格填写器**。它比你想象中聪明，给它原则就够了，别用模板把它的判断力读傻了。

### 它不会做的事

- ❌ 不会往提示词里塞无意义的格式模板
- ❌ 不会规定模型的思考流程（"先做A再做B最后做C"）
- ❌ 不会为凑字数而添加冗余描述
- ❌ 不会改变你的原始意图
- ❌ 不会留占位符（`[insert here]`、`TBD`）

## 安装

### Claude Code

```bash
claude skills install https://github.com/kkkkkkk123216/prompt-optimizer.git
```

### WorkBuddy

将 `SKILL.md` 和 `references/` 目录复制到：

```
~/.workbuddy/skills/prompt-optimizer/
```

最终目录结构：

```
prompt-optimizer/
├── SKILL.md              # 技能定义（触发条件 + 工作流 + 质量标准）
└── references/
    └── techniques.md      # 技术参考库（诊断维度 + 结构模式 + 反模式）
```

### 手动安装

```bash
git clone https://github.com/kkkkkkk123216/prompt-optimizer.git
```

将 `SKILL.md` 的内容作为 system prompt 的一部分粘贴到任何 AI 助手中即可。
